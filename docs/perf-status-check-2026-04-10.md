# Проверка статуса ускорений (2026-04-10)

## 1) Убрать N+1 на `/chats`
**Статус: сделано.**

В `GET /chats` используется один агрегирующий SQL-запрос через `prisma.$queryRaw` с `GROUP BY chatId`, который считает unread сразу по всем чатам пользователя.
Это заменяет подход с `Promise.all(...count...)` на каждый чат.

## 2) Не тянуть лишние поля/сущности при открытии списка чатов
**Статус: не сделано.**

`GET /chats` по-прежнему тянет `members` с вложенным `user` для каждого чата, плюс `messages` (last message) и `_count`.
То есть отдельного «легковесного» DTO для sidebar пока нет.

## 3) Виртуализация списка сообщений на фронте
**Статус: не сделано.**

Список сообщений в `ChatApp` рендерится прямым `.map(...)` по массиву сообщений.
Подключений `react-virtuoso`/`react-window` в коде не обнаружено.

## 4) Сократить «тяжесть» ChatApp
**Статус: частично сделано.**

- Компонент `ChatApp` всё ещё очень крупный и содержит много `useState`.
- Есть локальное выделение подкомпонентов (`Av`, `MediaAttachment`, `VoiceAttachment`), но нет выраженного разбиения на `MessageList/Composer/ChatHeader/TopicsBar` как отдельные контейнеры.
- `useCallback` применяется широко, но системного перевода тяжёлых блоков на `React.memo` не видно.

## 5) Кэш сообщений + SWR при открытии чата
**Статус: частично сделано.**

В `ChatContext` есть in-memory кэш сообщений с TTL `20_000` мс и защита от дублей запросов (`pending`/`pendingMore`).
Однако паттерн SWR в явном виде (показать кэш и параллельно тихо рефетчить) не реализован: при свежем кэше метод просто возвращает cached-данные без фонового обновления.

## 6) Дебаунс/батчинг `loadChats()` из сокет-событий
**Статус: сделано частично/в основном сделано.**

Есть `scheduleLoadChats(delayMs = 300)` с `setTimeout` и сбросом предыдущего таймера, что устраняет дребезг повторных reload.
Плюс часть событий обрабатывается точечными patch-обновлениями в state (`setChats` без полного рефетча).

## 7) Пагинация старых сообщений в UI
**Статус: сделано.**

- Бэкенд поддерживает cursor pagination (`cursor`, `hasMore`, `nextCursor`, `limit`).
- Клиент использует `loadMoreMessages` и догружает историю при прокрутке вверх (`onMessagesScroll` при `scrollTop < 120`).
- Новые старые сообщения prepend-ятся в начало массива (`[...older, ...existing]`).

## 8) Батчинг создания уведомлений в WS `message:send`
**Статус: сделано (обновлено 2026-04-10).**

В `ws/socket.ts` отправка уведомлений больше не выполняется строго последовательно для каждого участника
через `for ... await`.

Теперь:
- участники режутся на батчи по `50`;
- каждый батч выполняется параллельно через `Promise.allSettled`;
- в рамках одного батча одновременно создаются записи уведомлений и отправляются socket-ивенты.

Это уменьшает latency отправки сообщения в крупных чатах и убирает линейный рост времени ответа
по числу участников в худшем сценарии.

## 9) Режимы выдачи `GET /messages` (lite/full)
**Статус: сделано (обновлено 2026-04-10).**

В `GET /api/chats/:chatId/messages` добавлен query-параметр `view`:
- `view=full` (по умолчанию) — полная модель сообщения с `media` и `reactions`;
- `view=lite` — облегчённая модель для ленты: без тяжёлых include-коллекций `media`/`reactions`,
  но с `_count` по ним, а также базовыми полями, `from` и `replyTo`.

Это позволяет снижать payload и нагрузку на сериализацию в сценариях, где полный payload не нужен сразу.
